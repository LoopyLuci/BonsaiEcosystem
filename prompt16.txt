## 3. USOS Core – Absolute Minimal Kernel

The USOS core must provide **only the minimum set of primitives** required to run the Omnisystem. All other functionality is implemented as userspace services. This ensures the kernel is small, verifiable, and free of bloat.

### 3.1 Kernel Responsibilities (Implemented in Titan, Verified by Axiom)

- **Memory Manager**:
  - Physical memory allocation (page frames).
  - Virtual memory with paging, copy‑on‑write, and shared memory.
  - Capability‑based memory regions: a process can only access memory for which it holds a capability token.
  - No built‑in filesystem or device drivers (except a minimal boot console).

- **Process & Thread Scheduler**:
  - Preemptive multitasking with priority queues.
  - Earliest Deadline First (EDF) for real‑time tasks, plus a fair scheduler for normal processes.
  - The scheduler is formally verified to be deadlock‑free and to respect deadlines under admission control.

- **Inter‑Process Communication (IPC)**:
  - Synchronous and asynchronous message passing via capabilities.
  - A single, unified IPC mechanism used by all services. No special‑purpose syscalls for networking or files.

- **Capability System**:
  - All system resources (memory, IPC channels, I/O ports) are represented by capabilities.
  - Capabilities are unforgeable, transferable, and revocable. The kernel manages capability spaces per process.

- **Boot & Initial Process**:
  - The kernel loads a signed initial userspace image (the “Omni‑Service Manager”) from a CAS.
  - The service manager then starts all other services.

### 3.2 What the USOS Core Does NOT Include

These are implemented as userspace services:

- Filesystem (VFS, disk drivers, etc.)
- Networking stack (TCP/IP, UDP)
- Device drivers (except a minimal boot console)
- GUI / windowing system
- Compression, encryption libraries
- Database, blockchain, AI inference
- Language runtimes (Titan, Sylva, etc.)
- Package manager or build system

By keeping the kernel minimal, we make it auditable and formally verifiable, while allowing userspace services to be updated independently.

## 5. Build System & Self‑Hosting Toolchain

The entire Omnisystem must be **self‑hosting**: built by the Omni‑languages themselves.

### 5.1 `omni` Command

A single binary `omni` (written in Titan) that provides:

- `omni build [target]` – builds the whole system or a specific service.
- `omni test` – runs UBVM test suites.
- `omni run` – launches USOS (in a VM or on bare metal) and starts all services.
- `omni package` – creates a signed, content‑addressed system image for deployment.
- `omni repl` – starts the Sylva REPL (or Titan if specified).

### 5.2 Bootstrapping

Initially, the Titan compiler is built using the existing Rust compiler (in the old repository). Once the Titan compiler is mature enough, it is used to compile itself. The bootstrap chain:

1. Rust compiles the first Titan compiler (`titan0`).
2. `titan0` compiles the full Titan standard library and the self‑hosting compiler.
3. From that point, `titan` can rebuild itself entirely.

All Omni‑languages are then compiled by Titan (or by themselves).

### 5.3 Dependency Management

Dependencies are content‑addressed (using BLAKE3). The `omni.toml` file lists dependencies by hash, not by name, ensuring reproducibility. Dependencies can be fetched from the local CAS or from the `omni-p2p` mesh.

## 7. Replacing All 750+ Languages

The Omni‑languages will become the **only languages** used for new development in the ecosystem. Legacy code support is provided through transpilers (built with the existing BPLIS/LAIR infrastructure), but these are transitional.

### 7.1 Transpilation Strategy

- For the top 100 languages (C, C++, Rust, Python, JavaScript, Go, Java, etc.), we provide high‑fidelity transpilers to Titan. These are integrated into `omni import`.
- For the remaining 650+ languages, transpilers are generated automatically from the Polyglot Pong data (since we have a canonical Pong implementation in each language). The process: 
  1. Take the canonical Pong in language L.
  2. Use BPLIS to convert it to LAIR.
  3. Generate Titan source from LAIR (using the ULB).
  4. Package the result as a library that emulates the original language’s standard library.
- Over time, developers are encouraged to rewrite performance‑critical parts directly in Titan, using the transpiled code as a reference.

### 7.2 Deprecation Timeline

- **Phase A**: Transpilers available; all new services are written in Omni‑languages.
- **Phase B**: The old repository is frozen; no new features are added in other languages.
- **Phase C**: Transpilers are maintained for legacy code but not actively developed.
- **Phase D**: Eventually, the 750+ other languages are no longer needed for Bonsai development; their tooling may be removed from the core distribution (but preserved for historical purposes).
