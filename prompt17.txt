## 4. Core Conversion Engine

The conversion engine translates a driver from its **source interface** (e.g., Linux `pci_driver`) to a **target interface** (e.g., UOSC capability‑based driver). It is built as a **recursive rewriting system**.

### 4.1 Rule Format

```titan
// udc/rule.ti
struct ConversionRule {
    name: String,
    source_pattern: IrPattern,
    target_template: IrTemplate,
    condition: Option<Expr>,
    verify: Option<ProofId>,
}
```

Rules are stored in a content‑addressed rule database and can be updated atomically.

### 4.2 Example Rule – Linux `readl` to UOSC `mmio_read`

```titan
rule LinuxReadlToUsos {
    source: call("readl", [arg])
    target: call("UOSC_mmio_read", [arg, size=4])
    condition: arg.type == *const u32
    verify: "rule_readl_soundness.ax"
}
```

### 4.3 AI‑Optional Function Mapping

If a driver uses a function that has no direct rule, the converter can invoke an **AI advisor** (optional, feature‑gated) to propose a mapping. The advisor runs in a shadow mode and its suggestions are only accepted if:

- Confidence > 0.9.
- The suggested code passes an Axiom safety proof (via the `verify` plugin).
- A human council (or automated policy) approves after a validation period.

## 4. Omni‑IR Lowering & Driver Analysis

Drivers written in any language (C, C++, Rust, Titan, etc.) are first converted to Omni‑IR using the existing BPLIS/LAIR pipeline (now ported to Titan). The converter then performs:

- **Resource extraction**: identifies all MMIO addresses, I/O ports, interrupt lines, and DMA buffers.
- **Call graph analysis**: traces all hardware access calls (e.g., `readl`, `ioremap`, `dma_alloc_coherent`).
- **Effect inference**: annotates which parts of the driver perform I/O, memory allocation, or synchronisation.

### 4.1 Resource Dependency Graph

The converter builds a graph of resource accesses:

```
READ(LCR) → WRITE(DLL)
WRITE(THR) → WAIT_UNTIL(LSR.THRE)
```

This graph is used to generate correct access sequences in the target driver, and to verify that no resource is accessed without proper initialisation.

## 5. Rule‑Based Conversion Engine

The heart of the UDC is the **deterministic rule engine**. Conversion rules are stored as content‑addressed blobs in the UPLD and can be hot‑reloaded atomically.

### 5.1 Rule Format

```titan
struct ConversionRule {
    name: String,
    source_pattern: IrPattern,     // e.g., call("readl", [arg])
    target_template: IrTemplate,   // e.g., call("UOSC_mmio_read", [arg, 4])
    condition: Option<Expr>,
    proof_id: Option<Hash>,        // Axiom proof of soundness
```

### 5.2 Example Rule

Linux `readl(addr)` → UOSC `mmio_read32(addr)`

```titan
rule ReadlToMmioRead {
    source: call("readl", [arg]),
    target: call("UOSC_mmio_read32", [arg]),
    condition: arg.type == *const u32,
    proof: "proofs/readl_to_mmio.ax"
}
```

### 5.3 Rule Application Strategy

The engine walks the IR bottom‑up, applying the first matching rule at each node. If no rule matches, the converter can:

- Apply a **generic default rule** (e.g., replace `ioread32` with `mmio_read32`).
- Invoke the **optional AI advisor** (see Section 6).
- Fall back to a **manual mapping request** logged to the `observability` service.

### 5.4 Rule Database Management

New rules can be added by the Bonsai Council (for Bonsai Ecosystem) or by any user with appropriate capability tokens. Rules are versioned and cryptographically signed, and a quorum of council members must approve a rule before it becomes active in the default rule set.

## 6. AI‑Optional Enhancement Layer

AI is used only for **suggesting** mappings when a direct rule is missing. All AI enhancements are feature‑gated (`--features ai-enhancements`) and clamped by the deterministic Arbiter.

### 6.1 Architecture

- A small, offline‑trained model (stored as a verified, quantised blob in CAS) observes the IR and the device specification.
- When a node has no matching rule, the model proposes a translation.
- The proposal is **validated** against a safety envelope (e.g., memory bounds, correct register widths) by the Axiom proof engine.
- Only if the proposal passes verification is it applied. Otherwise, the converter falls back to a manual request.

### 6.2 Shadow‑Mode Validation

New AI suggestions are logged and compared against later manually‑written rules. After a statistically significant number of correct predictions, the AI can be promoted to “active advisor” by a council vote.

### 7.1 UOSC Native Driver (Capability‑Based)

```titan
// Generated UOSC driver (output of UDC)
pub struct UartDriver {
    mmio_cap: Capability,
    irq_cap: Capability,

impl Device for UartDriver {
    fn init(&mut self, caps: Vec<Capability>) -> Result<(), Error> {
        self.mmio_cap = caps.into_iter().find(|c| c.ty == "mmio").unwrap();
        self.irq_cap = caps.into_iter().find(|c| c.ty == "irq").unwrap();

    fn write_char(&self, c: u8) -> Result<(), Error> {
        self.mmio_cap.write32(THR_OFFSET, c as u32)
    }

    fn read_char(&self) -> Result<u8, Error> {
        while self.mmio_cap.read32(LSR_OFFSET) & LSR_DATA_READY == 0 {}
        self.mmio_cap.read32(RBR_OFFSET) as u8
    }
}
```

### 7.2 Linux Kernel Module Output

For Linux, the converter generates C code that uses the standard kernel APIs (`ioread32`, `request_irq`, etc.). This output can be compiled with the Linux kernel build system and loaded as a module.

### 7.3 Userspace Driver (UIO / VFIO / UOSC‑native)

For isolation, the driver can be emitted as a **userspace process** that maps MMIO regions via the UOSC `mmap` capability or Linux UIO, with interrupts delivered via eventfd.

### 7.4 Capability Manifest Generation

Alongside the driver, the converter produces a **signed capability manifest** that lists exactly which hardware resources the driver needs. The UOSC capability system enforces this manifest at load time, preventing any unauthorised access.

## 8. Formal Verification of the Converter

Using Axiom, we prove that every conversion rule preserves **observable device behaviour**. The proof uses a **simulation relation** between the source IR and target IR.

### 8.1 Rule Soundness Theorem

For each rule, we prove:

```ax
theorem rule_soundness(rule: ConversionRule):
    forall source_state: DeviceState, input: Input,
        let (new_state, output) = interpret(source_ir, source_state, input)
        let (target_state, target_output) = interpret(target_ir, source_state, input)
        output == target_output && new_state == target_state
```

This ensures that the target driver behaves identically to the source driver for all possible inputs.

### 8.2 Engine Safety Proofs

- **No deadlock**: the rule application loop always terminates.
- **Determinism**: the same driver + the same rule set always produces identical output.
- **Resource safety**: generated drivers never access resources outside their manifest.

These proofs are stored in the `udc/proofs/` directory and verified by the UVM on every commit.
