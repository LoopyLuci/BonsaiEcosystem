## 3. Detailed Design

### 3.1 Omnisystem Guest Kernel Modifications

The Omnisystem (UOSC) is a native kernel that expects to run on bare metal. To run it inside a Sanctum vault, we add a **hypervisor‑aware boot path**:

- **Guest mode**: Detect that the kernel is booting as a guest via a boot parameter (e.g., `kernel.guest=1`).
- **Use virtualised CPU features**: Instead of raw paging and interrupts, use the hypervisor’s interface (e.g., via a small shim that emulates hardware).
- **Paravirtualised drivers**: Instead of direct device access, the guest uses **virtio** drivers for console, block, network, and a special control channel.

**Minimal hypervisor shim**: A tiny (<10K LOC) component that:
- Sets up VMCS (or equivalent) to run the guest kernel in ring 0 but with intercepts.
- Handles privileged instructions (CPUID, HLT, I/O) and page faults.
- Forwards virtio requests to host.

Alternatively, we can use **UOSC’s existing ability to run as a “kernel module” inside another kernel** (as a co‑OS). This is already designed: UOSC can be loaded as a **process** (like a library OS) on top of a microkernel, using its own address space and capabilities. This is simpler than full virtualization.

**Preferred approach**: **Library OS mode** – The Omnisystem is compiled as a position‑independent binary that can be loaded into a Sanctum vault’s memory, then the vault’s execution context transfers control to it. The Omnisystem then uses the host’s system calls via a **capability‑based ABI** (provided by Sanctum). This is the most efficient and secure.

**Thus**: No hypervisor needed. The vault runs a single process that is the Omnisystem itself, with direct access to the host kernel’s capability interface. The Omnisystem kernel becomes a **process image** that runs in user space but with its own memory management. This is analogous to how `libos` works.

### 3.2 Sanctum Vault Configuration

Sanctum already provides a way to create a vault with:

- Isolated memory region (size configurable).
- Set of capabilities (for I/O, devices, networking).
- Entry point (start function) and initial register state.

We extend the vault to load an **Omnisystem binary** (the kernel + bundled initrd). The vault’s start function calls `uosc_main` with a context that includes:

- Boot info (memory map, device tree).
- A shared memory region for virtio queues.
- Handles to virtio devices (console, block, network).

### 3.3 Communication Channel (Host ↔ Guest)

To interact with the Omnisystem (send commands, receive logs, transfer files), we need a reliable, high‑bandwidth channel.

**Design**: Use **virtio‑console** for simple serial I/O, and **virtio‑user** or a custom **shared memory ring** for high‑speed data transfer.

- **Console**: Used for logging, panic output, and basic command shell.
- **Control channel**: A separate virtio‑queue that sends structured messages (protobuf or JSON) for operations like:
  - `RunCommand(cmd)` → execute a Sylva script or Titan program inside the guest.
  - `TransferFile(path, data)` → copy a file into the guest filesystem.
  - `GetMetrics()` → retrieve performance data.
  - `HotReload(path)` → trigger atomic update of a kernel module or service.

The host side (Bonsai) implements a client library (`bonsai‑guestctl`) that interfaces with these queues.

### 3.4 Bonsai Integration

Bonsai provides a CLI/API to manage vaults:


Internally, these commands use the **control channel** to talk to the guest’s `guest‑agent` (a small Titan service running inside the Omnisystem).

### 3.5 Guest Agent (Inside Omnisystem)

To facilitate host communication, the Omnisystem includes a **guest agent** service that:

- Runs as a privileged process.
- Listens on the virtio‑control queue.
- Parses requests and executes them (e.g., `RunCommand` → invokes Sylva REPL, captures output, returns result).
- Provides file system access via capability‑mediated APIs.

The agent is written in Titan and uses the same effect system as the rest of the Omnisystem.

## 1. Guest Architecture: From Library OS to Capability‑Native Guest

Instead of a traditional hypervisor or even virtio‑based paravirtualization, the Omnisystem runs as a **capability‑native guest** directly on the host’s Sanctum‑managed capability system. No device emulation is necessary.

### 1.1 Capability Passthrough

The host (Bonsai) grants the Omnisystem guest a set of **kernel capabilities** that map directly to host resources. The guest kernel (UOSC) is compiled with a `guest` feature that causes it to:

- Use the host’s physical memory capability to allocate pages (no separate page tables; the host’s MMU and the guest’s internal allocator share the same virtual address space through capability‑mediated memory regions).
- Receive interrupt notifications via a capability‑based IPC channel (no simulated interrupt controllers).
- Access block storage and network through shared‑memory rings protected by capabilities (no emulated PCI devices).

This eliminates all emulation overhead. The guest achieves **native instruction execution** with only a capability check on each system resource access.

### 1.2 Unified Capability Space

The Sanctum vault exports a **capability table** that the guest uses exactly like its own kernel’s capability table. For example, when the guest creates a new process, it requests a memory region capability from the host via a dedicated IPC call. The host’s Sanctum manager verifies the guest’s root capability and grants a sub‑capability. The guest never sees the host’s full capability space; it only holds a bounded set of derived capabilities.

This design is formally proven to enforce isolation: the guest cannot forge capabilities, and the host can revoke them at any time.

## 2. Dynamic Resource Scaling

A bleeding‑edge guest must be able to **grow and shrink** its resources without restarting.

### 2.1 Memory Hot‑Plug via Capability Resize

When the guest needs more memory, it calls the host through a `ResizeMemory` capability. The host, if allowed, extends the vault’s memory region and grants a new, larger memory capability. The guest’s kernel then hot‑plugs the additional pages into its allocator, updating its page tables atomically (using the Omnisystem’s own atomic update mechanism). Shrinking works similarly: the guest compacts and returns a sub‑capability.

### 2.2 CPU Core Hot‑Plug

The guest can request additional virtual CPUs (which map to host threads or cores) via a `SpawnVCpu` capability. The host spawns a new thread that runs the guest’s idle loop, and the guest’s scheduler immediately begins dispatching work to it. This is particularly valuable for Aether actor systems that need to scale with load.

### 2.3 Network Bandwidth Throttling

The guest’s network capability has a bandwidth limit. The host can adjust this limit in real time, enabling Service‑Level Agreements (SLAs) for multi‑tenant scenarios.

## 3. Live Migration

Omnisystem guests can be **migrated between physical hosts** while running, without dropping connections or losing state. This is possible because:

- The guest’s entire state (CPU registers, memory, capabilities) is captured as a **deterministic snapshot** using the kernel’s `memfork` syscall.
- The snapshot is transferred via the `p2p` service to the target host.
- The target host creates a new vault and restores the snapshot, resuming execution from the exact instruction.

Migration can be triggered manually or automatically (e.g., for load balancing). The UVM continuously validates that migrated instances produce identical outputs to non‑migrated ones, ensuring determinism.

## 4. Deterministic Execution Across Vaults

To make the guest fully deterministic (for testing and reproducibility), we leverage the Omnisystem’s **effect system** and **Universal Validation Mesh**.

- **Deterministic mode**: When the guest is booted with `--deterministic`, the host provides a fixed PRNG seed and disables any non‑deterministic interrupts (e.g., network IRQs are replaced by a recorded trace). The guest agent logs all effect invocations to the `observability` service.
- **Replay**: A previous guest run can be replayed bit‑for‑bit by loading its initial snapshot and feeding the recorded input trace. The UVM uses this to verify that OS updates do not introduce regressions.
- **Shadow execution**: A second guest instance can run in “shadow mode” where its outputs are compared against the live instance; any divergence triggers an alert.

- The `p2p` service (running on the host or as a dedicated actor) routes messages between vaults using the existing post‑quantum, multi‑path bonded transport.
- This enables distributed Omnisystem clusters to run entirely within isolated vaults, forming a **virtual planet‑scale mesh** that mirrors the physical deployment.

## 6. Formal Verification of the Vault Boundary

The interface between host and guest must be proven secure. Using Axiom, we verify:

- **Capability integrity**: The guest can never access a resource for which it does not hold a valid capability.
- **Confinement**: The guest cannot observe or influence other vaults or the host beyond its granted capabilities.
- **Liveness**: The host’s capability‑grant mechanism never deadlocks, and the guest cannot starve the host.
- **Resource bounds**: The guest cannot consume more CPU, memory, or I/O than its capabilities allow.

These proofs are integrated into the UVM CI pipeline, ensuring the vault manager remains correct across updates.

## 7. Hot‑Reload of Guest Kernel and Services

The Omnisystem’s atomic hot‑reloading is extended to work across the vault boundary.

- **Service hot‑reload**: The host sends a new service binary through the control channel. The guest agent atomically swaps the service’s code in memory (using the kernel’s generation counter), drains in‑flight requests, and activates the new version without dropping connections.
- **Kernel hot‑reload**: A new kernel image can be loaded into a separate memory region and, after verification, the guest can `kexec` (or similar) into it, preserving all capabilities and open connections. The host verifies the new kernel’s signature before allowing the transition.

This allows continuous deployment of Omnisystem updates without any downtime.

## 3. Detailed Design – Recursive Bonsai

### 3.1 How a Bonsai Instance is Represented Inside a Vault

A Bonsai instance is **not** a full virtual machine – it is a **set of processes** running on top of a microkernel (UOSC) that provides:

- Capability‑based security.
- Isolated address spaces for each service.
- IPC between services.

When we “run Bonsai inside a Sanctum vault”, we:

1. Create a Sanctum vault (host side) with a dedicated memory region.
2. Load the **UOSC kernel binary** (compiled for library OS mode) and the **initial Bonsai userspace image** (a CAS‑stored initrd).
3. Transfer control to the kernel’s entry point, passing a boot context that describes:
   - Memory regions (RAM, device memory).
   - Virtio device handles (console, control, block, net).
   - Capabilities for I/O (granted by host).
4. The kernel boots and starts the Bonsai service manager, which in turn launches the core services (p2p, storage, etc.) and the Bonsai CLI.

### 3.2 Nested Sanctum Vaults (Bonsai inside Bonsai inside Bonsai…)

To support nesting, the **guest Bonsai** must itself have the ability to create Sanctum vaults. This requires:

- The host to delegate the necessary hardware virtualization capabilities (e.g., VT‑x, AMD‑V, or CHERI compartment creation) to the guest.
- The guest’s UOSC kernel to be able to call host hypercalls (or use a paravirtualised interface) to create a new vault.
- The host to enforce resource limits (e.g., a guest cannot allocate more than its assigned memory).

**Implementation approach**:

- **Paravirtualised “vault manager”**: The host exposes a virtio‑vault device that the guest can use to request new vault creation. The host then creates a new Sanctum vault (as a child of the guest’s context) and returns a handle.
- **Two‑level security**: The host ensures that the guest cannot escalate privileges; it only allows nested vaults if the hardware supports nesting (e.g., VMX with nested paging) and within the guest’s quota.

Thus, a Bonsai instance can run `bonsai vm create` inside its own CLI, which triggers a virtio request to the host, which in turn creates a new vault. This yields a **recursive tree** of Bonsai instances.

### 3.3 Resource Management & Quotas

To prevent a guest from consuming excessive resources, the host assigns:

- **Memory limit**: Hard cap (e.g., 1 GiB).
- **CPU limit**: Share of cores (e.g., 0.5 core) or real‑time quota.
- **Storage limit**: Maximum disk space for its initrd and writable overlay.
- **Network bandwidth**: Throttled via virtio‑net QoS.

The host monitors usage and can kill a misbehaving guest.

### 3.4 Communication Between Host and Guest (and Between Nested Guests)

We use **virtio** (paravirtualised I/O) for:

- `virtio-console`: Guest console output (logs, shell).
- `virtio-control`: A custom ring for high‑level commands (run CLI commands, transfer files, get metrics).
- `virtio-block`: Guest root filesystem.
- `virtio-net`: Network (if needed).
- `virtio-vault`: For nested vault creation (custom device).

The host side provides a library (`libbonsai-guest`) that the host Bonsai CLI uses to interact.

### 3.5 Guest Bonsai Modifications

The Bonsai Ecosystem (when built as a guest) must be compiled as a **position‑independent, library‑OS‑aware** image. Specifically:

- The **UOSC kernel** must be built with `CONFIG_GUEST_MODE=y` (disables direct hardware access, uses virtio instead).
- The **Bonsai service manager** must detect that it is running inside a guest and use the virtualised devices instead of hardware drivers.
- The **Bonsai CLI** must be able to call the virtio‑vault device to spawn nested instances.

All of this is already possible because Bonsai/UOSC is capability‑based; the host simply provides a different set of capabilities to the guest.

- **Multi‑tenant platform**: Each user gets their own nested Bonsai instance.
- **Training & education**: Each student can have an isolated Bonsai playground.
- **Driver & service testing**: Test kernel modules or services inside a nested instance, safely.

## 7. Limitations & Future Work

- **Nesting depth**: Limited by hardware support (e.g., Intel VT‑x nested page tables). May be shallow.
- **Performance for deep nesting**: Each level adds overhead. Optimization: use library OS mode and minimize hypercalls.
- **GPU passthrough**: Requires SR‑IOV or mediated passthrough; not trivial.
- **Real‑time guarantees**: Nested virtualisation may interfere with real‑time scheduling.
