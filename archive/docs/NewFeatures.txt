## Executive Summary

To achieve truly production‑grade, robust, and intelligent background services across UOSC, Omnisystem, and the Bonsai Ecosystem, the system must adopt **deterministic‑first, capability‑based, and hot‑reloadable** service architecture. ## 1. Core Principles for Background Services

| Principle | Implementation |
|-----------|----------------|
| **Demand‑activated** | Services are not always running; they are spawned when a capability is requested and deactivated after a timeout or explicit command. |
| **Snapshotable state** | Every service can be paused, serialized to a content‑addressed CAS snapshot, and restored later (like a VM snapshot but for a single process). |
| **Capability‑based activation** | A client must hold a capability to request a service; the kernel’s capability table tracks which services are available. |
| **Deterministic replay** | The entire lifecycle of a service (spawn → work → snapshot → restore) can be recorded and replayed for debugging and audit. |
| **Hot‑reloadable** | Service binaries can be updated atomically without stopping the service; the snapshot mechanism ensures state continuity. |
| **Resource‑controlled** | Each service runs in its own Sanctum vault with resource limits (memory, CPU, I/O). |

## 3. Service Lifecycle – From Spawn to Archive

### 3.1 Service Manifest (UMS Module)

Each service is packaged as a UMS module with a manifest that includes:

```json
{
  "name": "fax",
  "binary": "blake3:abcd...",
  "capabilities_required": ["USB", "NET"],
  "snapshot_supported": true,
  "idle_timeout_secs": 300,
  "max_memory_mb": 256,
  "max_cpu_percent": 50
}
```

### 3.2 Demand‑Activated Spawn

When a client (e.g., Bonsai Buddy) sends a request to the SLM with a capability to the `fax` service, the SLM:

1. Checks if a snapshot of that service exists (in the client’s context or globally).
2. If yes, it restores the service vault from the snapshot (including memory, register state, and open capabilities).
3. If not, it spawns a new vault from the service binary, passes the required capabilities (e.g., `USB`), and starts the service.
4. The client receives a connection capability to the service.

The client never knows whether the service was already running – the SLM abstracts that.

### 3.3 Idle Timeout & Snapshot

The SLM monitors service activity. If a service receives no requests for its configured `idle_timeout_secs`, the SLM:

1. Sends a `pause` signal to the service (the service saves its internal state to a buffer).
2. The kernel snapshots the entire vault (memory, registers, open capabilities) – using the same `memfork` syscall as live migration.
3. The snapshot is compressed with UCE and stored in CAS, with a reference in the service’s manifest.
4. The vault is destroyed.
5. Any future request to the same service (with a matching capability) triggers a restore from that snapshot.

### 3.4 Archive & Deep Sleep

After a longer period (e.g., 24 hours) or when the system is under memory pressure, the SLM can **archive** a snapshot: move it to a slower CAS tier (e.g., network storage) and delete the local copy. A restore will fetch it on demand.

### 3.5 Restore & Continue

Restoration is atomic:

1. The SLM loads the snapshot from CAS.
2. It creates a new Sanctum vault and restores the memory image.
3. The kernel reconstructs the capability table and re‑establishes IPC ports.
4. The service resumes execution exactly where it left off – no state loss, no re‑initialisation.

This works because the service was written to support pausing (it can flush pending writes and release resources that cannot be snapshotted). The UOSC kernel’s snapshot mechanism captures everything else.

## 4. Bonsai Buddy as a Standalone Agent

### 4.1 What is Bonsai Buddy?

Bonsai Buddy is a **mobile/desktop companion app** that interacts with the Omnisystem. It can run independently – on a phone, a laptop, or an edge device – without requiring the full Omnisystem stack. It uses the same Service Lifecycle Manager but can also operate in **offline mode** with local snapshots.

### 4.2 Architecture of Standalone Buddy

Bonsai Buddy is a **Tauri desktop app** (or a mobile app) that embeds:

- A lightweight **SLM client** (not the full SLM).
- A local **cache of UMS modules** (for services it needs).
- A **local CAS** (smaller, on‑device) for snapshots.
- A **TransferDaemon node** (optional, for peer‑to‑peer sync).

When the Buddy needs a service (e.g., to scan a document using the `scan` service), it:

1. Checks if it is connected to a full Omnisystem cluster (over VPN or local network). If yes, it forwards the request to the cluster’s SLM.
2. If not (offline mode), it spawns a **local Sanctum‑like process** (on macOS/Windows, using the hosted‑light adapter) and runs the service from the UMS module. The state is snapshotted locally.
3. When connectivity is restored, the Buddy can synchronise snapshots with the cluster (pushing local changes and pulling remote ones).

### 4.3 Snapshot Synchronisation (CRDT‑based)

Service state is represented as a CRDT (Conflict‑Free Replicated Data Type). When two Buddies or a Buddy and a cluster both modify the same service, the snapshots can be merged deterministically. This is implemented by:

- Storing the service state as a set of immutable updates (event sourcing).
- Using a Merkle tree to detect divergence.
- Applying the same merge algorithm that the service uses internally (e.g., for a CRDT‑based todo list).

Thus, the Buddy can work offline and later sync – a true mobile experience.

## 5. Production‑Grade Service Implementation Details

### 5.1 Deterministic Service Design

Every background service must be **deterministic**: given the same sequence of inputs (requests) and the same initial state, it must produce the same outputs and the same internal state changes. This is enforced by:

- Using the kernel’s deterministic random number generator (fixed seed when snapshotting).
- Avoiding calls to `gettimeofday` (use the deterministic virtual clock provided by the kernel).
- Using the `observability` service to record all non‑deterministic inputs (e.g., user input, sensor data) – these are part of the snapshot.

### 5.2 Snapshot‑Aware Service Programming

The service must implement a simple interface:

```titan
pub trait Snapshotable {
    // Prepare to be snapshotted: flush buffers, release non‑persistent resources
    fn on_pause(&mut self) -> Result<Vec<u8>, Error>;
    // Resume after restore: given the saved state, reconstruct internal structures
    fn on_resume(&mut self, state: &[u8]) -> Result<(), Error>;
}
```

The SLM calls these methods before taking the kernel snapshot. This is the only integration point.

### 5.3 Health Monitoring & Self‑Healing

The SLM itself is supervised by an Aether supervision tree. If the SLM crashes, it restarts and re‑loads the service registry from the audit log. Service vaults are not affected because they are separate processes.

The SLM also monitors service health via heartbeats. If a service becomes unresponsive, it restarts it from the last good snapshot.

### 5.4 Resource Control & Quotas

Each service vault has a resource capability that limits:

- Memory (hard limit, enforced by kernel).
- CPU (via `cpu_quota` capability).
- I/O bandwidth (via network or disk capabilities).
- Snapshot storage (maximum number of snapshots, total size).

The SLM enforces quotas before spawning.

### 5.5 Observability & Audit

All service lifecycle events (spawn, snapshot, restore, archive, crash) are logged to the `audit-log` service. The Universe can replay the entire history of any service.

The UVM runs periodic “chaos” tests: it randomly pauses and restores services, injects network delays, and verifies that the service’s state remains consistent.

2. Service Lifecycle Manager (SLM)

The SLM is an Aether actor with a capability to spawn Sanctum vaults. It is the single orchestrator for all background service lifecycles.

2.1 SLM Responsibilities

· Maintain a registry of available service manifests (fetched from UMS).
· Listen for capability‑mediated service requests from clients.
· Spawn service vaults on demand, or restore them from a snapshot if one exists.
· Monitor service activity; trigger pause‑and‑snapshot after an idle timeout.
· Archive old snapshots to slower storage; purge based on quota.
· Restore services from snapshots on re‑request, with full state continuity.
· Enforce resource quotas (memory, CPU, I/O) before spawning.
· Log every lifecycle event to the audit‑log for observability and replay.

2.2 SLM Internal State

The SLM maintains a map of service_instance_id → ServiceInstance:

```
ServiceInstance {
    id: u64,
    manifest_hash: [u8; 32],
    vault_id: Option<u64>,          // currently running vault, if any
    snapshot_hash: Option<[u8; 32]>, // latest snapshot in CAS
    capabilities: Vec<Capability>,
    state: ServiceState,            // Running, Paused, Archived, Failed
    last_access: u64,               // timestamp of last client request
    resource_usage: ResourceUsage,
}
```

The SLM itself is supervised by an Aether supervision tree; if it crashes, it restarts and reloads its state from the audit‑log (which is also snapshotted periodically).

2.3 Service Manifests (UMS Module)

Every background service is packaged as a UMS module with a manifest that declares:

The SLM fetches the module from UMS, verifies its council signature, and stores the manifest in its registry.

3. Demand‑Activated Spawn & Restore

When a client (e.g., Bonsai Buddy) wants to use a service, it sends a request to the SLM with the appropriate capability. The SLM executes the following deterministic state machine.

3.1 Spawn Decision Logic

```
Input: (client_cap, service_name)
1. Look up service manifest by name.
2. Check if a snapshot exists for this service in the client's context (or globally).
3. If snapshot exists:
   a. Verify snapshot integrity (BLAKE3 hash).
   b. Create a new Sanctum vault.
   c. Restore the memory image and capability table from the snapshot.
   d. Resume execution from the saved instruction pointer.
4. If no snapshot:
   a. Create a new Sanctum vault.
   b. Load the service binary from UMS.
   c. Grant the required capabilities.
   d. Start the service process.
5. Record the new ServiceInstance in the SLM state.
6. Return a connection capability to the client.
```

3.2 Snapshot Mechanism (Kernel‑Level)

The UOSC kernel provides a memfork syscall extended to support single‑vault snapshots:

```titan
pub fn snapshot_vault(vault_id: u64) -> Result<[u8; 32], Error>;
```

This call:

· Pauses all threads in the vault.
· Writes the entire memory image (code, data, stack, heap, register state) to a temporary buffer.
· Serialises the vault’s capability table (including which capabilities are held and their current state).
· Compresses the image using the Universal Compression Engine (UCE) and writes it to CAS.
· Returns the BLAKE3 hash of the snapshot.

Restoration is the inverse:

```titan
pub fn restore_vault(snapshot_hash: &[u8; 32], caps: Vec<Capability>) -> Result<u64, Error>;
```

The kernel creates a new vault, decompresses the memory image, loads it, reconstructs the capability table, and sets the instruction pointer to the saved value. The service resumes exactly where it was paused.

3.3 Service‑Aware Pausing

Services implement the Snapshotable trait (a Titan trait):

```titan
pub trait Snapshotable {
    /// Called before the kernel snapshot. Flush buffers, close transient connections.
    fn on_pause(&mut self) -> Result<Vec<u8>, Error>;
    /// Called after restore. The opaque `state` is what `on_pause` returned.
    fn on_resume(&mut self, state: &[u8]) -> Result<(), Error>;
}
```

The SLM invokes these methods before calling snapshot_vault and after restore_vault. The opaque state can contain, e.g., file offsets, TCP sequence numbers, or any other data that cannot be captured by a raw memory dump.

4. Idle Timeout & Automatic Snapshot

The SLM monitors the last_access timestamp of each running service. A background actor (the idle‑watcher) periodically checks for services that have exceeded their idle_timeout_secs.

When a service becomes idle:

1. The SLM sends a pause command to the service.
2. The service’s on_pause is called.
3. The SLM invokes snapshot_vault.
4. The resulting snapshot hash is stored in the SLM state; the old snapshot (if any) is archived (moved to cold CAS storage or deleted if quota exceeded).
5. The vault is destroyed.
6. The SLM logs the event.

If a client requests a service that is currently Paused or Archived, the SLM restores it transparently—the client never knows the service was not running.

Bonsai Buddy is a Tauri desktop/mobile application that functions as a local SLM client. It embeds:

Online mode: The Buddy is connected to an Omnisystem cluster (via VPN, local network, or the internet). Service requests are forwarded to the cluster’s full SLM; the Buddy acts as a thin client.

Offline mode (no connectivity): The Buddy’s local SLM handles everything. It spawns services locally using the hosted‑light adapter (or a local Sanctum‑like environment on the host OS). All state is snapshotted to the local CAS.

Hybrid mode: The Buddy is online but the requested service is not available in the cluster (or the user wants local processing). The Buddy runs the service locally but can synchronise its snapshot with the cluster later.

5.2 Snapshot Synchronisation via CRDTs

For services that need to synchronise state across multiple Buddy instances (or between a Buddy and a cluster), the service state is modelled as a CRDT. The snapshot contains a Merkle tree of the service’s event log. When two snapshots are merged, the CRDT merge algorithm deterministically resolves any conflicts.

The Buddy can push its local snapshot to the cluster’s SLM, and the cluster can reply with a merged snapshot that includes updates from other Buddies. This is exactly how offline‑first apps (like a distributed todo list) work.

6. Deterministic Service Design Requirements

Every background service must be deterministic by default. This means:

· All I/O is mediated by the effect system; the service can only perform effects that are declared in its manifest.
· The kernel’s deterministic random number generator is used (seeded from the snapshot’s hash).
· The virtual clock (provided by the kernel) is used instead of gettimeofday.
· All non‑deterministic inputs (user gestures, sensor data, network packets) are recorded in the snapshot’s event log. During replay, these inputs are fed back from the log.

The UVM can then replay the entire lifecycle of a service and verify that the outputs are identical—this is the ultimate test of correctness.

7. Resource Control & Quotas

The SLM enforces resource limits per service:

· Memory: hard limit enforced by the kernel’s capability system. If the service exceeds it, the kernel kills the vault and the SLM restores from the last snapshot.
· CPU: quota capability; the scheduler ensures the service does not exceed its share.
· I/O: network and disk bandwidth limits via capabilities.
· Snapshot storage: maximum number of snapshots (e.g., 5) and total size (e.g., 1 GB). When the quota is exceeded, the oldest snapshot is archived or deleted.

3. Restores the service from the last good snapshot.
4. Logs the incident.

### Step 1: Extend Kernel with Single‑Vault Snapshot Syscalls

**Current state:** `memfork` snapshots the entire system (all processes).  
**Required change:** Add two new syscalls:

```c
// kernel/syscall.h
int64_t snapshot_vault(uint64_t vault_id, uint8_t* out_hash);   // returns 0 on success, -1 on error
int64_t restore_vault(const uint8_t* snapshot_hash, uint64_t* out_vault_id);
```

**Implementation in Titan (kernel module):**  
- Iterate through the vault’s memory regions (already tracked in the kernel’s capability table).  
- Serialise them into a temporary buffer; compress using UCE; write to CAS.

## 3. Model Lifecycle – From Training to Production

### 3.1 Offline Training (Custom Model Building Framework)

The **Model Building Framework** (MBF) is a Titan/Sylva toolkit that:

- Collects deterministic execution traces from the UVM (kernel logs, service metrics, performance counters).
- Anonymises and aggregates traces.
- Trains small models (e.g., linear regression, decision trees, small neural networks) for specific tasks:  
  *Next page pre‑fetcher* (storage), *load balancer* (scheduler), *actor placement* (Aether), *congestion predictor* (network), *energy‑aware scaling* (power manager).
- Produces models in a deterministic format (e.g., decision trees, lookup tables, or ONNX graphs with deterministic operators).
- Outputs a **model UMS module** containing the model weights and a manifest with input/output schemas and safety envelopes.

**Key requirement**: The training process is deterministic (fixed random seeds, same input → same model). The model itself must be **deterministic** – given the same input, it must produce the same output (no dropout, no stochastic sampling at inference time).

### 3.2 Shadow Mode Validation

Before a model can advise production, it must run in **shadow mode** for a statistically significant period (e.g., 10,000 predictions). During shadow mode:

- The deterministic core runs as usual.
- The model’s suggestions are logged but **never applied**.
- The UVM compares the model’s suggestions against the actual decisions made by the deterministic core, and measures:
  - **Safety**: Would the model have caused a violation of safety envelopes? (must be 0%).
  - **Efficacy**: Would the model have improved performance (latency, throughput, energy) compared to the deterministic baseline?
  - **Confidence**: The model’s output confidence (if available) and consistency over time.
- After passing the validation period, the model can be promoted to **advisory mode** by a council‑signed policy.

## 4. Safety Envelopes (Axiom‑Proved)

Every model output must be **clamped** by a safety envelope that is formally verified with Axiom. For example:

- **Prefetch model** – The predicted block number must be within the valid range of the file system. The kernel’s prefetch function caps it.
- **Scheduler load balancer** – The suggested CPU allocation cannot exceed the total available cores or starve real‑time tasks.
- **Actor placement** – The suggested node must be online and have sufficient capacity; otherwise, the deterministic fallback (round‑robin) is used.

Axiom proves:

```ax
theorem safety_envelope_enforced:
  forall model_output: Output, envelope: SafetyEnvelope,
    let clamped = apply_envelope(model_output, envelope) in
    clamped.value ≥ envelope.min ∧ clamped.value ≤ envelope.max
```

These proofs are attached to the model module and checked at load time.

4. Safety Envelope System — Axiom‑Proven Bounds

The safety envelope is a Titan library that provides per‑model clamping functions. Each envelope is parameterized by the model's manifest and is formally verified.

4.1 Envelope Structure

```titan
pub struct SafetyEnvelope {
    pub min_output: f64,
    pub max_output: f64,
    pub min_confidence: f32,
    pub max_latency_us: u64,
    pub resource_limits: ResourceLimits,
}

pub fn clamp_output(raw: ModelOutput, envelope: &SafetyEnvelope) -> Option<ModelOutput> {
    if raw.confidence < envelope.min_confidence { return None; }
    let clamped_value = raw.value.max(envelope.min_output).min(envelope.max_output);
    Some(ModelOutput { value: clamped_value, confidence: raw.confidence })
}
```

     clamped.unwrap().value ≤ env.max_output)

  forall raw: ModelOutput, env: SafetyEnvelope,
    clamp_output(raw, env) == clamp_output(raw, env)  // idempotent

Shadow mode is the mandatory validation phase before any model can influence production decisions.

5.1 Shadow Instance

5.2 Validation Metrics

The UVM continuously computes:

Metric Computation Threshold
Safety Violation Rate % of suggestions that would have violated safety envelope Must be 0%
Efficacy Delta (AI suggestion benefit) − (deterministic outcome benefit) Must be > 0 for promotion
Consistency Variance of outputs for similar inputs Must be below threshold
Availability % of requests answered within timeout Must be > 99.9%

5.3 Promotion Process

1. Shadow model accumulates MIN_VALIDATION_SAMPLES (default: 10,000) predictions.

3. The report is submitted to the governance council.
4. Council votes (threshold signature) to promote the model to active.
5. The Orchestrator updates the model's state; future requests use the model's suggestions (still clamped by the envelope).

5.4 Automatic Demotion

If an active model's safety violation rate exceeds 0% at any time (detected by the UVM's continuous monitoring), it is immediately demoted to shadow mode, and the deterministic core takes over. The incident is logged and the council is notified.

fn extract_features(trace: Trace) -> Vec<TrainingExample> {

}
