### OS Runtime Model (How OSes Run Inside Omnisystem)

```
OMNISYSTEM PERSPECTIVE:
Each OS appears as a special "super-process" with:
├── Dedicated Vault (Sanctum Vault)
├── Assigned vCPUs (virtual processors)
├── Allocated Memory (virtual address space)
├── Virtual Devices (storage, network, GPU)
├── Isolation Boundaries (cryptographic)
├── Security Policies (what it can access)
└── Resource Limits (fair share allocation)

FROM OS PERSPECTIVE:
Each OS "thinks" it's running on bare metal:
├── Has virtual CPUs (appears as multicore)
├── Has virtual memory (appears as full address space)
├── Has virtual devices (appears as real hardware)
├── Makes system calls (intercepted by OmniOS)
├── Does I/O operations (virtualized)
├── Manages processes (within its vault)
└── Handles interrupts (virtual interrupts)

OMNISYSTEM INTERCEPTS:
├── All system calls
├── All I/O operations
├── All memory access (page faults)
├── All interrupts (virtual exceptions)
├── All hardware access (device operations)
└── All inter-OS communication

RESULT:
Perfect isolation + High performance + Transparent operation
```

### Vault Isolation Mechanisms

```
1. MEMORY ISOLATION
   ├── Separate page tables per vault
   ├── SMMU/IOMMU protection
   ├── Encrypted memory (if hardware supports)
   ├── No cross-vault memory access
   └── Automatic memory cleanup on vault termination

2. FILE SYSTEM ISOLATION
   ├── Per-vault file system namespace
   ├── Encrypted storage (per vault)
   ├── No shared file system access
   ├── Virtual device mapping per vault
   └── Automatic snapshot capability

3. NETWORK ISOLATION
   ├── Per-vault virtual NIC
   ├── Virtual network switch
   ├── Per-vault network policies
   ├── Firewall rules per vault
   └── Inter-vault communication (controlled)

4. DEVICE ISOLATION
   ├── Device access whitelist per vault
   ├── GPU passthrough or emulation per vault
   ├── USB device pass-through with restrictions
   ├── Serial device assignment
   └── Automatic device cleanup

5. CRYPTOGRAPHIC ISOLATION
   ├── Unique encryption key per vault
   ├── AES-256-GCM encryption (at rest)
   ├── TLS 1.3 for inter-vault communication
   ├── Perfect forward secrecy
   └── Regular key rotation

6. PROCESS ISOLATION
   ├── All processes run within vault
   ├── No escape to hypervisor
   ├── Process syscall filtering
   ├── Capability-based restrictions
   └── Automatic resource limits
```

### Vault Lifecycle Management

```
VAULT STATES:
Created → Initialized → Running → Paused → Stopped → Destroyed

VAULT CREATION:
1. Allocate vault ID

4. Create file system namespace
5. Create network namespace
6. Initialize security policies
7. Ready for OS installation

VAULT INITIALIZATION:
1. Copy OS installation media

VAULT RUNNING:

2. System calls intercepted
3. I/O virtualized
4. Memory managed
5. Processes scheduled
6. Resources monitored

VAULT OPERATIONS:
├── Pause (suspend to disk)
├── Resume (restart from pause)
├── Checkpoint (create snapshot)
├── Restore (revert to snapshot)
├── Migrate (move to different hardware)
├── Clone (duplicate vault)
├── Backup (full vault backup)
└── Restore (restore from backup)

VAULT TERMINATION:
1. Shutdown OS gracefully
2. Save state (if desired)
3. Deallocate resources
4. Clear memory
5. Delete encryption keys
6. Clean file system

```
