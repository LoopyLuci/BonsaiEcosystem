## 2. AriaDB: The Data Engine (Detailed)

### 2.1 Storage Engine: Hybrid LSM‑Forest + Bw‑Tree

AriaDB’s storage is a **log‑structured merge tree** with adaptive page sizes (4KB–256KB), persistent memory (PMem) for hot rows, and NVMe/S3 for cold data. It borrows from WiscKey, RocksDB, and FASTER.

| Layer          | Technology                               | Purpose                                         |
|----------------|------------------------------------------|-------------------------------------------------|
| **Memory Table** | Cuckoo hash + ART (adaptive radix tree)  | Lock‑free, PMem‑backed, fast ingest            |
| **Versioned Blocks** | Copy‑on‑write, zstd compressed      | Immutable L0‑L2, time‑travel enabled           |
| **Columnar Store** | Parquet‑like, SIMD‑aware               | Analytical queries, predicate pushdown         |
| **Persistent Log** | AOF + LSN indexing, multi‑raft        | Durability, replication, point‑in‑time recovery |

**Key innovations (merged):**
- **Learned indexes** (RMI, PGM‑index) replace B‑trees for read‑heavy column families.
- **Adaptive storage** – hot rows stay in PMem, cold rows move to NVMe, archival to S3 with automatic promotion.
- **Row stitching** – 128‑bit UUIDs with locality hints enable distributed pointers without cross‑shard joins in hot path.
- **Schema‑aware encoding** – dictionary compression, delta coding, and run‑length encoding based on column type.

### 2.2 Multi‑Model Implementation

All logical models (relational, document, graph, time‑series, vector, geospatial) are compiled into the same physical LSM tree using a **unified type algebra**:

```
type User = {
  id: Uuid @pk,
  name: String @index(fulltext),
  friends: Set<Ref<User>> @edge(follows),
  profile: Json,                   // flexible document
  last_seen: Timestamp,
  embedding: Vector<768> @index(hnsw)
}
```

- **Relational** → primary key columns + secondary indexes as separate LSM trees.  
- **Document** → JSON stored as a binary blob with optional partial indexes (bloom filters on paths).  
- **Graph** → adjacency lists encoded as sorted sets of `(from, to, edge_properties)` in a dedicated column family; traversal operators compile to range scans.  
- **Time‑series** → ordered by timestamp, with automatic rollup (materialized views) and retention policies.  
- **Vector** → HNSW index stored as a separate LSM tree that stays consistent via write‑ahead logging.

### 2.3 Immutability, Branching & Time‑Travel

AriaDB uses a **content‑addressed, append‑only** storage model. Each commit produces a new root hash (Merkle DAG). This gives:

- **Zero‑cost snapshots** – branch by copying the root pointer.
- **Git‑like workflows** – `CREATE BRANCH feature_x FROM production`; work on isolated copy; `MERGE` with automatic conflict detection (three‑way merge on values, with user‑defined resolution for complex types).
- **Time‑travel queries** – `SELECT * FROM users AS OF '2025-01-15T10:00:00Z'` is a simple root lookup.
- **Forensic lineage** – every cell stores the transaction ID and wall‑time, recursively linkable.

### 2.4 Distributed Consistency: Raft + Calvin + CRDTs

AriaDB supports **per‑transaction consistency levels**:

| Mode                     | Mechanism                                                                 | Use case                                  |
|--------------------------|---------------------------------------------------------------------------|-------------------------------------------|
| **Strict Serializable**  | Calvin‑style deterministic ordering + pre‑declared read/write sets       | Financial transactions, inventory         |
| **Snapshot Isolation**   | MVCC + SIREAD bitmap conflicts                                           | Most OLTP                                 |
| **Causal**               | Hybrid logical clocks (HLC) + vector clocks                              | Multi‑region, low‑latency reads           |
| **Eventual (CRDT)**      | State‑based CRDTs (counters, maps, sets, registers, lists)               | Collaborative editing, counters, presence |

**Implementation details:**
- **Metadata cluster** – Raft with 3–5 nodes, stores shard mapping, schema versions, capabilities.
- **Data sharding** – two‑layer: consistent hash on partition key, range shards per node for locality.
- **Distributed transactions** – Percolator 2.0 (async commit, parallel 2PC) adapted for sub‑100ms p99 across 3 AZs.
- **Contention handling** – optimistic for low conflict (<5% abort), pessimistic row‑level latches (MCS queue) for hotspots.

### 2.5 Reactive & Incremental Engine (Differential Dataflow)

Every query, view, and aggregation is compiled into a **dataflow graph** of operators (map, filter, join, group, window). The runtime maintains the graph incrementally using **Differential Dataflow** techniques:

- **Collections** are versioned multisets (bags) with timestamps.
- **Changes** flow as `(add, remove, modify)` batches.
- **Operators** compute deltas by propagating changes through partial derivatives (e.g., `count` sees only inserts/deletes).
- **Latency** – sub‑millisecond for small changes, millisecond for large ones.

Example in Aria language:

```aria
let active_posts = query {
  p in Post where p.published && p.author.active
}
// active_posts is a LiveSet<Post> – automatically updated
active_posts.observe(|delta| { /* UI updates */ })
```

The engine also supports **fixed‑point iterations** (e.g., PageRank, graph reachability) with incremental convergence.

### 2.6 Vector & AI Integration (Expanded)

- **Vector indexes** – HNSW, IVF, DiskANN built into the LSM; automatically updated on writes.
- **Model serving** – `CREATE MODEL sentiment FROM 'huggingface://distilbert'`; call inside expressions:  
  `SELECT sentiment(review.text) FROM reviews WHERE sentiment(review.text).label = 'positive'`.
- **Feature store** – Declare feature pipelines:  

  ```aria
  feature user_activity_7d(user: Ref<User>) -> Vector<10> {
    let posts = Post.where(author == user && created_at > now() - 7d);
    [count(posts), avg(posts.likes), ...]
  }
  ```
  The engine materialises features online (for inference) and offline (for training) with point‑in‑time correctness.
- **Homomorphic encryption** – optional for regulated workloads; operators on encrypted vectors run inside secure enclaves (SGX/TDX).

## 3. The Aria Language (Detailed)

### 3.1 Syntax & Type System

Aria is a **functional‑first, statically typed** language with a syntax resembling a cleaner mix of Rust, Swift, and TypeScript. Key features:

- **Dependent refinement types** – `Int where value >= 0`, `String where len <= 280`.
- **Linear types** – resources that cannot be dropped without explicit disposal (file handles, DB connections, transactions).
- **Effect tracking** – pure vs. `Db`, `Net`, `Random`, `Time`, `Panic`. Effects are part of the function signature.
- **Option / Result** – no `null`, no unchecked exceptions. `Option<T>` and `Result<T, E>` with pattern matching.

Example:

```aria
fn divide(a: Int, b: Int) -> Int / DivByZero {
  if b == 0 { raise DivByZero } else { a / b }
}

fn safe_divide(a: Int, b: Int) -> Option[Int] {
  match divide(a, b) {
    case DivByZero -> None
    case v -> Some(v)
  }
}
```

### 3.2 First‑Class Queries (No ORM)

Queries are built‑in expressions that return **live collections** and are checked against the current schema at compile time.

```aria
// Schema defined in Aria – canonical source of truth
model Blog {
  type User {
    id: Uuid = Uuid::gen(),
    name: String,
    email: String @unique,
    posts: List<Ref<Post>>,
  }
  type Post {
    id: Uuid,
    title: String,
    body: String,
    published: Bool,
    author: Ref<User>,
  }
}

// Parameterised query, returns a reactive set
fn recent_posts_by_user(user_id: Uuid, limit: Int) -> LiveSet[Post] {
  query {
    p in Post where p.author.id == user_id && p.published
    order by p.created_at desc
    take limit
  }
}
```

The compiler translates the comprehension into an AriaDB dataflow plan. There is no string concatenation, no injection risk, and no N+1 problem – joins are automatically optimised using indexes.

### 3.3 Algebraic Effects & Handlers

All side‑effects are modelled as **algebraic effects**. Handlers can replace the real implementation with mocks, logs, or alternative backends.

```aria
effect Db {
  fn query_one<T>(q: Query<T>) -> T / NotFound
  fn execute(q: Mutation) -> ()
}

// A function that performs database work
fn update_user_email(id: Uuid, new_email: String) -> () / Db {
  let user = Db.query_one(User.where(id == id))  // may raise NotFound
  user.email = new_email
  Db.execute(user.update())
}

// Testing: swap real Db with in‑memory handler
let test_handler = handler {
  fn query_one<T>(q) -> T / NotFound { mock_db.lookup(q) }
  fn execute(q) -> () { mock_db.apply(q) }
}
handle update_user_email(uuid, "test@example.com") with test_handler
```

The runtime uses this mechanism to provide **transactional scopes** with automatic retries and idempotency.

### 3.4 Actors for Stateful Services

Actors encapsulate state and are **persistent** – their state is stored in AriaDB and survives crashes.

```aria
actor ShoppingCart {
  state {
    items: Map[Ref[Product], Int],
    owner: Ref[User],
  }

  fn add_item(prod: Ref[Product], qty: Int) {
    state.items = state.items.merge({prod: qty})
  }

  fn checkout() -> Order {
    transactional {
      let order = Order.create(state.owner, state.items)
      state.items = Map::empty()
      order
    }
  }
}
```

Actors are location‑transparent; the runtime can move them across nodes. Each actor processes messages sequentially (no locks). The state is automatically checkpointed to AriaDB after every transactional change.

### 3.5 Reactive UI Binding

The standard library includes a **declarative UI framework** that binds directly to `LiveSet` and `Live` types.

```aria
view ProfilePage(user_id: Uuid) -> Element {
  let user = User.get(user_id)      // returns Live[Option[User]]
  let posts = recent_posts_by_user(user_id, 10)  // LiveSet[Post]

  Column[
    match user {
      None -> Text("Loading...")

        Text(u.name),
        Text("Email: " + u.email),

}
```

When the underlying data changes (e.g., a new post is published), only the affected `PostCard` re‑renders. The UI runtime communicates with AriaDB via a binary WebSocket protocol that streams deltas.

### 3.6 Schema Evolution as Code

Migrations are Aria functions, versioned and tested alongside application logic.

```aria
migration add_user_avatar (version 2, depends_on 1) {
  // Add a nullable field
  alter type User {
    avatar_url: Option[String] = None
  }
  // Backfill data
  for user in User.all() {
    user.avatar_url = Some(generate_avatar_url(user.email))
    user.update()
  }
}
```

The toolchain:
- Computes the minimal set of changes (no full table rewrite if possible).
- Generates a dual‑version schema so that old and new code can run simultaneously during rollout.
- Supports online `ALTER` operations (adding a column with default is instant).

# Part 1: The Database Engine — **AriaDB**

AriaDB is a **poly‑paradigm, temporal, reactive state fabric** with a unified multi‑model storage kernel and a JIT‑compiled query executor. It replaces traditional SQL databases, document stores, graph databases, caches, and vector stores in a single system.

## 1.1 Storage Engine: LSM‑Forest + Bw‑Tree Hybrid

The storage engine is built for modern hardware (NVMe, persistent memory, RDMA) and adapts its data structures online.

```
┌───────────────────────────────────────────────────────────┐
│  Memory Table (Cuckoo + ART index, lock‑free, PMem‑aware) │
├───────────────────────────────────────────────────────────┤
│  Versioned Blocks (Copy‑on‑write, zstd compressed)       │
│  └── Row group metadata (min/max, bloom filters,         │
│       learned min‑max predictors)                        │
├───────────────────────────────────────────────────────────┤
│  Columnar Secondary Store (PAX / Parquet‑like, SIMD scan)│
├───────────────────────────────────────────────────────────┤
│  Persistent Write‑Ahead Log (AOF, LSN‑indexed,           │
│   multi‑Raft replication)                                │
└───────────────────────────────────────────────────────────┘
```

**Key innovations:**
- **Adaptive page sizes** (4 KB–256 KB) – tuned by online reinforcement learning based on access patterns.
- **Learned indexes** – Recursive Model Indexes (RMI) and PGM‑index replace B‑trees on read‑heavy column families, reducing index memory by 90%.
- **Tuple routing with temperature awareness**:
  - *Hot* rows reside in Intel Optane PMem (byte‑addressable, persistent).
  - *Warm* rows on NVMe SSDs.
  - *Cold* rows automatically archived to S3‑compatible object storage, with lazy promotion on access.
- **Row stitching** – distributed pointers are 128‑bit UUIDs that embed a locality hint (shard + node id) to avoid cross‑shard joins in the hot path.
- **Zero‑copy** transfer between storage and query execution via `io_uring` and registered buffers.

## 1.2 Query Execution: Vectorized JIT with Adaptive Reoptimisation

Queries are compiled to machine code using **Cranelift** (or LLVM) with explicit SIMD (AVX‑512, NEON) and run as native functions.

```rust
// Internal plan representation
Plan::HashJoin(
    left: Plan::Filter( predicate>18, Plan::Scan("users") ),
    right: Plan::Scan("orders"),
    on: (left.col("id"), right.col("user_id"))
)
// Compiles to a tight loop with AVX‑512 gather/scatter for hash table probes.
```

**Execution pipeline:**
- **Push‑down of predicates, projections, and aggregations** to the storage layer, where columnar chunks are scanned with SIMD and early pruning via zone maps.
- **Adaptive reoptimisation** – after 10k rows the runtime compares actual vs. estimated cardinalities; if off by >2×, it transparently re‑compiles the rest of the query with a better plan.
- **Hybrid transactional/analytical (HTAP)** – the same storage engine serves both OLTP and OLAP queries; resource isolation is enforced with cgroups‑based throttling per query class.
- **Consistent snapshots** for long‑running analytical queries via MVCC (multi‑version concurrency control) with hybrid logical clocks (HLC). No read locks block writes.

## 1.3 Concurrency & Transactions

AriaDB employs a **multi‑modal transaction manager** that dynamically chooses the optimal path:

- **Fast path (OCC + timestamp ordering)** for low‑contention workloads:
  - Transactions execute speculatively against a consistent snapshot.
  - At commit, a **serializable snapshot isolation (SSI)** check detects read‑write and write‑write conflicts via SIREAD bitmaps kept in the transaction context.
  - Abort rate is kept <5% by back‑pressure.
- **Pessimistic fallback** for high‑contention hotspots:
  - Automatic contention detection (via per‑row conflict counters) escalates to lightweight MCS‑queue‑based latches on individual rows.
- **Deterministic execution** for multi‑shard transactions:
  - Inspired by Calvin, transactions that pre‑declare their read/write sets are globally ordered and executed with zero communication overhead beyond the ordering layer.
- **Distributed commit** uses a **Percolator‑style async 2PC** with parallel prewrite and commit phases, optimized for <100 ms p99 latency across three availability zones.
- **Transaction prioritisation** – short transactions (<1 ms) jump ahead of long scans, preventing head‑of‑line blocking.

## 1.4 Distribution & Consistency

**Two‑layer sharding**:
- Top layer: consistent hashing over a large virtual node space (100k+ shards) for load distribution.
- Within each node: range shards for locality and efficient scanning.

**Metadata & ordering**:
- A **Raft‑based metadata cluster** manages shard assignments, schema versions, and topology.
- For strict serialisability, a global sequencer issues monotonically increasing timestamps; for cross‑shard deterministic transactions, a **Calvin‑style sequencer** pre‑orders based on read/write sets.

**Multi‑model consistency** in the same cluster:
- **Strict serialisability** for OLTP (banking, inventory).
- **Causal consistency with CRDTs** for collaborative, multi‑master scenarios (real‑time docs, shared whiteboards). Built‑in CRDT types: counters, registers, maps, sets, ordered lists.
- **Eventual consistency** for search and analytics, with user‑defined conflict resolution policies.

**Read repair** happens automatically using monotonic read timestamps; anti‑entropy is performed by background Merkle tree comparison.

## 1.5 Data Model: Temporal, Multi‑Model, Conflict‑Aware

All data is managed through a single type system. Schema definitions specify storage behaviour:

```aria
// Schema defined in Aria, compiled into AriaDB
type User {
    id: Uuid @pk,
    email: String @unique,
    name: String @index(fulltext),
    age: Int where value >= 0,
    profile: Json,                           // flexible document
    embedding: Vector<768> @index(hnsw(ef=200, M=16)),
    created_at: Timestamp = now(),
    
    // Temporal: keep every version forever
    @temporal(retention = forever)
    
    // CRDT map for concurrent modifications
    @crdt(type = "map")
}
```

- **Entities** (tables/collections) can be marked `temporal = true` to enable point‑in‑time queries by default.
- **Conflict‑free** settings (`@crdt`) activate automatic merging of concurrent writes.
- **Graph edges** are declared via `Set<Ref<T>>` with traversal hints (`@edge(type = "follows")`).
- **Vector types** natively support HNSW, IVF, and DiskANN indexes, refreshed incrementally on writes.
- **Time‑to‑live** (`_ttl`) auto‑expires records after a defined period.

## 1.6 Integrated AI & Vector Capabilities

- **Model serving** inside the database: `CREATE MODEL spam_filter FROM 'onnx_model.onnx'` and use directly in expressions.
- **Automatic feature stores** – point‑in‑time correct materialisations for training and inference.
- **Privacy‑preserving operators** – optional homomorphic encryption for aggregations and secure multi‑party computation for multi‑party joins.

# Part 2: The Programming Language — **Aria**

Aria is a **high‑level, statically‑typed, functional‑first language** that compiles to native code and WebAssembly. It treats the database as an integral part of the type system, not an external resource.

## 2.1 Type System with Full Database Integration

- **Dependent refinement types** for database constraints: `String where len <= 280`, `Int where value >= 0`. These are enforced by the compiler using SMT solvers, with runtime checks at system boundaries.
- **Linear types** for resources: `Transaction`, `FileHandle`, `DbConnection` must be consumed exactly once (committed or rolled back, closed).
- **Effect system** to track side effects: `DbRead`, `DbWrite`, `Io`, `Alloc`, `Panic`. Functions are annotated; pure functions cannot touch the database or I/O.
- **Gradual typing** – rapid prototyping is untyped; production code is fully typed with no `any`.

**Example of schema‑aware, effect‑tracked code:**

```aria
fn active_users(region: String) -> List<User> 
    effects: [DbRead]
{
    // The `query` block is checked against the User schema at compile time
    let q = query {
        u in User
        where u.region == region && u.active
        order by u.name
    };
    conn.execute(q)  // returns List<User>, no casting, no ORM
}
```

## 2.2 First‑Class Queries, Reactive Views, and Streaming

Queries are **expressions**, not strings. They produce typed, **reactive collections** that update automatically:

```aria
// A live view that stays in sync with the database
let top_posts: LiveSet<Post> = query {
    p in Post
    where p.published
    order by p.likes desc
    limit 10
};

// Subscribe to deltas
top_posts.observe(|delta, full_set| {
    // delta: {added: List<Post>, removed: List<Post>, modified: List<Post>}
    render(full_set);
});
```

**Streaming** with windowed aggregations is built into the language:

```aria
let stream = query {
    from o in Order
    where o.placed_at > now() - 1.hour
    group by o.region, tumbling_window(1.minute) into g
    select { region: g.key, orders: count(g.rows), total: sum(g.rows.amount) }
    emit changes every 10.seconds
};
for await let window in stream {
    dashboard.update(window);
}
```

The compiler translates all of this into an incremental dataflow graph that runs natively inside AriaDB, pushing deltas to the application with sub‑millisecond latency. No polling, no change‑data‑capture pipelines.

## 2.3 Algebraic Effects & Error Handling

Side effects are managed with **algebraic effect handlers**, enabling testability and safe resource use:

```aria
effect Db {
    fn execute<T>(q: Query<T>) -> T;
}

fn transfer(tx: linear Transaction, from: Ref<Account>, to: Ref<Account>, amount: Decimal) 
    -> linear Transaction 
    effects: [DbWrite]
{
    let acc = tx.get(from);
    if acc.balance < amount {
        return tx;  // caller must rollback
    }
    tx.update(from, {balance: acc.balance - amount});
    tx.update(to,   {balance: acc.balance + amount});
    tx.commit()     // consumes the linear tx
}
```

**Error handling** uses effect handlers, not exceptions; there is no `null`:

```aria
fn divide(a: Int, b: Int) -> Int throws DivisionByZero {
    if b == 0 { throw DivisionByZero }
    else { a / b }
}

let result = try divide(10, 0) {
    DivisionByZero => 0   // handle
};
```

All resource management is checked at compile time – you cannot forget to commit a transaction or close a connection.

## 2.4 Actor Model & Persistence

Stateful components are modelled as **persistent actors**, each with private transactional state that survives restarts:

```aria
actor ShoppingCart for user: UserId {
    state {
        items: Map<Ref<Product>, Int>,
        last_touch: Timestamp,
    }
    
    fn add(product: Ref<Product>, qty: Int) effects: [DbWrite] {
        state.items = state.items.merge({product: qty});
        state.last_touch = now();
    }
    
    fn checkout() -> Order {
        transactional {
            let order = Order::create(state.items);
            state.items = Map::empty();
            order
        }
    }
}
```

Actors are location‑transparent; the runtime handles message routing, concurrency (single‑threaded per actor, no locks), and periodic state snapshots. This replaces microservice boilerplate.

## 2.5 Structured Concurrency & Capabilities

**Concurrency** is structured and safe, inspired by Swift’s task groups but with database‑aware scheduling:

```aria
fn load_profiles(ids: List<UserId>) -> List<Profile> 
    effects: [DbRead]
{
    let results = await* for id in ids {
        spawn fetch_profile(id)   // bounded parallelism
    };
    results
}
```

- The runtime uses an **M:N scheduler** with work stealing, backed by `io_uring` (Linux) or IOCP (Windows).
- **Capability‑based security** – no ambient authority. The `main` function receives only the capabilities it needs:

```aria
fn main(db: Capability["app_db"], logger: Capability["log"]) {
    // cannot open files or sockets unless passed a capability
}
```

Inside the database, capabilities extend to **row‑level security** – Aria functions define policies that are compiled into query predicates, ensuring that even direct DB access respects access control.

## 2.6 Memory Management

Aria uses a **hybrid memory model**:

- **Region‑based allocation** (arena) for function‑scoped objects – memory is freed automatically on scope exit. The compiler proves most allocations are region‑bound, eliminating tracing overhead for ~90% of all objects.
- **Tracing GC with zero‑pause guarantees** for long‑lived heap data. Aria implements a C4‑inspired concurrent collector with read barriers and incremental compaction.
- **Inline storage** – objects ≤ 128 bytes are embedded directly in their parent, avoiding pointer indirection.

## 2.7 UI & Reactive Data Binding

Aria includes a declarative UI framework that directly subscribes to live data:

```aria
fn user_page(id: Uuid) -> View {
    let user = User.get(id);   // Live<Option<User>>, reactive
    Column {
        Text(user.name),
        ForEach(user.posts, |post| PostCard(post))
    }
}
```

`User.get(id)` returns a live reference that updates when the underlying data changes. The UI re‑renders only the minimal affected parts, driven by the same incremental engine that powers materialized views. The entire stack—from disk to pixel—is reactive.

# Part 3: The Unified Query Layer (AriaQL)

Queries are written in Aria itself, but for tooling, data exploration, and legacy interop we provide a standalone **AriaQL** syntax that is a 1:1 textual representation of the same AST:

```sql
-- Standalone AriaQL (for CLI, BI tools)
FROM User
WHERE region = "EU" AND active = true
AGGREGATE BY country
SELECT {
    country: country,
    cnt: count(*),
    avg_age: avg(age)
}
AS OF '2026-06-01T00:00:00Z'   -- time travel
EMIT STREAM (tumbling window 10 minutes);
```

This is exactly equivalent to the Aria embedded query:

```aria
query {
    from u in User
    where u.region == "EU" && u.active
    group by u.country into g
    select { country: g.key, cnt: count(g.rows), avg_age: avg(g.rows.age) }
}
```

**Key enhancements over traditional SQL:**
- **Composable** – every query is an expression that can be reused, parameterised, and stored as a variable.
- **First‑class streaming** – windows, watermarks, and exactly‑once semantics are built in.
- **Versioned reads** – `AS OF` timestamp is a native concept, not a hack.
- **Graph traversals** – `TRAVERSE 1..3 (follows|blocks) WHERE …` with pruning and aggregation.
- **Recursive queries** – native fixed‑point semantics (Datalog‑style) for graph analytics and ML iteration.
- **No NULL** – `OPTIONAL` fields are expressed via `Option<T>` with pattern matching, eliminating entire classes of bugs.

# Part 4: Production Hardening

## 4.1 Observability & Debugging

- **eBPF probes** attached to every query, actor, and transaction, providing zero‑overhead latency histograms, error rates, and lock contention maps.
- **OpenTelemetry native** – spans are automatically created for all operations, with `traceparent` propagation across services.
- **Continuous profiling** (perf + async stack unwinding) integrated into the runtime; p99 latencies can be traced down to exact CPU instruction stalls.
- **Time‑travel debugger** – pause the system, step through actors, inspect the full database state at any historical point, and even branch to replay a scenario with modified inputs.

## 4.2 Security & Compliance

- **Capability‑based access** – applications receive only the privileges they are explicitly given; no ambient file system or network access.
- **Cryptographic verification** – schema changes, stored procedures, and even query plans are signed; nodes validate signatures before execution.
- **mTLS everywhere** – automatic SPIFFE‑style identity and TLS 1.3 encryption for all inter‑node and client communication.
- **Row‑level security** – policies written in Aria are compiled into query predicates, guaranteeing that users see only what they are allowed to.
- **Data loss prevention** – the planner can enforce geographical residency: `RESTRICT COLUMN credit_card TO region "EU"`.
- **GDPR/CCPA** – time‑travel makes it trivial to delete a user’s data from all versions with a single command, and the Merkle DAG proves erasure.

## 4.3 Operational Excellence

- **Online schema evolution** – multiple schema versions coexist during a rollout; no global locks, copy‑on‑write versioning.
- **Automatic materialized views** – defined as ordinary queries; the differential dataflow engine maintains them incrementally with sub‑millisecond freshness.
- **Cross‑region active‑active** – built on CRDTs for data conflicts and a human‑in‑the‑loop resolver for schema conflicts.
- **Chaos engineering built‑in** – inject latency, partitions, and node failures via a standard API for integration testing.
- **Zero‑downtime upgrades** – the storage format is binary compatible; rolling upgrades proceed without service interruption.

# Part 5: The Unified Stack as a Whole

In a typical Aria application, there is **no separation** between database, application server, and caching layer. A developer writes:

```aria
// Schema definition – this is the single source of truth
database MyApp {
    type User { ... }
    type Order { ... }
    // ... tables, indexes, policies
}

// Business logic – runs inside the database or application server transparently
fn place_order(customer: Ref<User>, cart: Cart) -> Order
    effects: [DbWrite]
{
    transactional {
        if customer.credit < cart.total {
            abort "insufficient credit"
        }
        let order = Order::create(customer, cart.items);
        customer.credit -= cart.total;
        order
    }
}

// UI – reactive, refreshed automatically
fn checkout_page(user: Ref<User>) -> View {
    let cart = user.active_cart; // Live<Option<Cart>>
    Button("Place Order", on_click = || place_order(user, cart.value))
}
```

The compiler:
1. Validates the entire program against the live database schema.
2. Compiles queries to JIT‑native code.
3. Packages the application as a single static binary (or Wasm module) that can run embedded with a full AriaDB node or connect to a distributed cluster.

# Part 6: Migration Path from SQL / PHP

1. **Bidirectional gateways** – AriaDB can expose PostgreSQL and MySQL wire protocols, translating legacy SQL into the internal representation on the fly.

4. **Dual‑write period** – write to both legacy and AriaDB, verify with audit queries, then switch reads.
5. **Phased rollout** – move one feature at a time, using the AriaQL console for ad‑hoc queries during transition.
