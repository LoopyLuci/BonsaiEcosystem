## Phase 2: Porting OpenCV to Rust (`cv` Crate)

The Rust port will follow a layered architecture, prioritizing memory safety and concurrency without sacrificing performance.

### Step 2.1: Core Rust Bindings via FFI
*   **Mechanism**: The widely adopted `opencv` crate auto-generates bindings for over 90% of the OpenCV 5 API by parsing C++ headers with `libclang`.

```rust
// Code for a simple FFI binding to OpenCV's `imread` function
use opencv::core::Mat;
use opencv::imgcodecs::{imread, IMREAD_COLOR};

fn load_image(path: &str) -> Result<Mat, opencv::Error> {
    let image = imread(path, IMREAD_COLOR)?;
    Ok(image)
}
```

### Step 2.2: Layered Architecture
The binding is structured in three layers:
1.  **FFI Layer**: The raw, unsafe C interface.
2.  **Binding Layer**: Unsafe Rust wrappers that manage memory and call the FFI functions.
3.  **High-Level Safe Rust Layer**: Provides an ergonomic, memory-safe API, leveraging Rust's ownership model to eliminate manual memory management.

### Step 2.3: Memory Safety & RAII
The `Mat` struct is wrapped to leverage Rust's RAII (Resource Acquisition Is Initialization). This guarantees memory is freed automatically when the `Mat` goes out of scope, eliminating a major class of memory errors.

### Step 2.4: Async/Await & Concurrency
*   **Integration**: The `cv` crate will integrate with the `rayon` data-parallelism library, allowing safe parallel iteration over image pixels. For asynchronous pipelines, it will interoperate with `tokio`.

### Step 2.5: Comprehensive Safety & Sanitizer Testing
*   **Tooling**: The test suite will be run under `miri` (Rust's interpreter for unsafe code) and sanitizers (ASan, TSan) to catch memory leaks, data races, and undefined behavior.

### Step 2.6: UMS Integration & Hot-Reload
*   **Process**: The final `cv` crate will be compiled into a dynamic library (`libcv.so`), signed, and published to the UMS, allowing for atomic hot‑reload.

## Phase 1: Universal Module System (UMS) Integration

The first concrete step is to make the original OpenCV 5 library available as a signed, content‑addressed UMS module. This enables immediate use from any Bonsai or Omnisystem component that can load native shared libraries.

### 1.1 Build Matrix and Multi‑Arch Compilation
A Titan build script (`opencv/build.ti`) compiles OpenCV 5 for every supported target:

| Target | Arch | OS |
|--------|------|----|
| `x86_64‑linux` | x86_64 | Linux (glibc) |
| `aarch64‑linux` | AArch64 | Linux |
| `riscv64‑linux` | RISC‑V 64 | Linux |
| `x86_64‑macos` | x86_64 | macOS |
| `aarch64‑macos` | Apple Silicon | macOS |
| `wasm32‑wasi` | Wasm | WASI |

The build script invokes CMake with the appropriate toolchain and produces a standard `libopencv_*.so` or `.dylib` for each module.

### 1.2 Module Manifests
Each architecture‑specific artifact is packaged into a **UMS module** with a manifest:

```yaml
name: opencv-core
version: 5.0.0
arch: x86_64-linux
exports:
  - symbol: cv::imread
    signature: "fn(String) -> Mat"
capabilities:
  - mem:256MB
  - cpu:avx2
```

The module is compressed with the **Universal Compression Engine (UCE)** and its BLAKE3 hash recorded.

### 1.3 Publishing and Council Signing
The module is signed by the Bonsai Council (5‑of‑7 threshold BLS signature) and pushed to the global UMS mesh. Once published, any Omnisystem node can fetch it via TransferDaemon.

### 1.4 Loading and Hot‑Reload
The `module‑loader` service verifies the signature, checks that the caller holds the required capabilities (e.g., `MEM`, `CPU`), and maps the library into the process. When a new version is published, the old one is drained and the new one atomically swapped—no restart needed.

## Phase 2: Rust Port (`cv` crate)

The Rust port provides a safe, idiomatic API while reusing the native OpenCV binaries as a transitional measure. It is packaged as a UMS module, enabling Bonsai and Omnisystem applications written in Rust to use OpenCV with full memory safety.

### 2.1 Layered Architecture
1.  **FFI Layer**: Raw, unsafe bindings generated by `bindgen` from the OpenCV 5 C++ headers.
2.  **Binding Layer**: Unsafe Rust wrappers that manage memory and translate error codes.
3.  **High‑Level Safe Layer**: Types like `Mat`, `UMat`, `Scalar` are wrapped in Rust structs with RAII, `Send`/`Sync` implementations, and iterator support.

```rust
// Example safe wrapper
pub struct Mat {
    inner: *mut opencv_sys::Mat,
}

impl Mat {
    pub fn new(rows: i32, cols: i32, typ: i32) -> Result<Self> {
        let inner = unsafe { opencv_sys::cv_create_mat(rows, cols, typ) };
        if inner.is_null() { Err(Error::Alloc) } else { Ok(Mat { inner }) }
    }
}

impl Drop for Mat {
    fn drop(&mut self) {
        unsafe { opencv_sys::cv_release_mat(self.inner); }
    }
}
```

### 2.2 Async and Parallelism
The `cv` crate integrates with `tokio` and `rayon`. Image processing pipelines can be expressed as asynchronous streams, and pixel‑wise operations are automatically parallelised.

### 2.3 Testing and Verification
All tests run under `miri` and ASan/TSan. The Rust implementation is validated against the canonical UVM test suite—every Rust function must produce bit‑identical output to the C++ reference for the same seed.

### 2.4 UMS Packaging
The final `cv` crate is compiled to a dynamic library (`libcv.so`), signed, and published as a UMS module. Its manifest declares capabilities (e.g., `MEM`, `CPU`), dependencies (the `opencv-core` native module), and exported symbols. It can be hot‑reloaded like the native module.

## Phase 3: Titan Rewrite (Systems Language)

Titan is the native systems language of the Omnisystem. We will rewrite the core computer vision primitives directly in Titan, eliminating all foreign dependencies. This is the definitive, sovereign implementation.

### 3.1 Rewrite Strategy
We rewrite modules incrementally, starting with `core` (the `Mat` class, basic operations) and then `imgproc`, `imgcodecs`, and `dnn`. Each rewritten module is a Titan crate that exports an `extern "uabi"` interface for cross‑language calls.

### 3.2 Titan Data Structures
The `Mat` is a capability‑managed buffer:

```titan
pub struct Mat {
    width:  usize,
    height: usize,
    channels: u8,
    depth:   u8,
    data:    Capability<[u8]>,   // linear memory capability
}
```

Memory is allocated via a capability, guaranteeing that the kernel enforces bounds and ownership. SIMD intrinsics (AVX‑512, SVE, Neon) are used for performance, with the compiler automatically selecting the best variant.

### 3.3 Hardware Abstraction via Capabilities
Functions that benefit from GPU acceleration (e.g., convolution) accept an optional `Gpu` capability. If present, the function dispatches to a GPU kernel; otherwise, it uses the CPU SIMD path. This is completely transparent to the caller.

```titan
pub fn gaussian_blur(img: &Mat, radius: u32, gpu: Option<GpuCap>) -> Result<Mat, Error> {
    if let Some(g) = gpu {
        gpu_gaussian_blur(img, radius, g)
    } else {
        cpu_gaussian_blur_simd(img, radius)
    }
}
```

### 3.4 Formal Verification of Algorithms
Key algorithms are accompanied by Axiom proofs:

```ax
theorem gaussian_blur_smoothing:
  forall img: Mat, radius: u32,
    let result = gaussian_blur(img, radius, None) in
    result.width == img.width ∧ result.height == img.height ∧
    (∀ x y, result.at(x,y) == weighted_avg(img, x, y, radius))
```

The Axiom proof is checked during compilation; any code change that violates the theorem is rejected.

### 3.5 UMS Integration
The Titan modules are compiled to native shared objects, signed, and published to the UMS. They carry a `proof‑certificate` manifest entry pointing to the Axiom proof, allowing consumers to independently verify correctness before loading.

## Phase 5: Aether Actor System for Distributed Vision

Aether enables planet‑scale, fault‑tolerant vision pipelines.

### 5.1 Actor Pipeline
A typical pipeline consists of actors:
- `FrameGrabber`: reads frames from a camera or video file.
- `Preprocessor`: resizes, normalises, converts colour space.
- `Detector`: runs a DNN model (e.g., YOLO).
- `Tracker`: follows objects across frames.
- `Logger`: records results to the `observability` service.

Actors communicate via Aether messages, which are automatically distributed across the mesh by TransferDaemon.

### 5.2 CRDT State
Shared state (e.g., the list of detected objects) is stored as a CRDT set. Concurrent detections from multiple cameras merge without conflicts, providing a unified view.

### 5.3 Supervision and Self‑Healing
The `Supervisor` actor monitors the pipeline. If the `Detector` actor crashes (e.g., GPU out‑of‑memory), it is automatically restarted on a different node with more GPU memory, using the capability system to allocate resources.

### 5.4 Integration
The Aether actors use the Titan or Sylva OpenCV modules through the Universal ABI. The pipeline is defined declaratively in a YAML file and deployed with a single command:

```bash
build pipeline deploy vision-pipeline.yaml
```
