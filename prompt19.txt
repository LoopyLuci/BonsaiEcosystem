"# Next‑Generation Environment Fabric: Unified Sanctum Vault Management for Emulations, Simulations, VMs, and Containers

## Executive Summary

The **Environment Fabric (EF)** is a unified, production‑grade system for creating and managing isolated execution environments – **emulations** (full system emulation, e.g., QEMU), **simulations** (deterministic network or hardware simulation, e.g., BUSH), **virtual machines** (hardware‑accelerated VMs), and **containers** (process‑level isolation) – all built on top of **Sanctum vaults**. The EF provides a single, intuitive CLI (`build env ...`) and a deterministic, capability‑based API. It integrates with the Omnisystem’s resource management, P2P fabric for live migration, deterministic replay, hot‑reloadable components, and optional AI‑predictive scaling. The design is formally verified for isolation and resource accounting.

| Principle | Implementation |
|-----------|----------------|
| **Unified primitive** | Every environment is a Sanctum vault (hardware‑isolated). The environment type determines the initial payload and device models. |
| **Deterministic by default** | Environments can be run in deterministic mode (fixed seed, record & replay) for reproducible testing. |
| **Capability‑based security** | Each environment receives a sub‑root capability that limits its resources and access. Nested environments derive sub‑sub‑roots. |

## 3. Environment Specification Schema

Environments are defined in YAML or TOML. Example:

```yaml

name: dev-vm
type: vm
description: "Development VM for Bonsai testing"
resources:
  memory: 2048    # MB
  cpus: 2
  storage: 10G    # root disk size
  quota:
    memory_soft: 2048
    memory_hard: 2560
network:
  mode: bridge    # bridge, isolated, nat, host
  interface: eth0
  ip: 192.168.1.100
  gateway: 192.168.1.1
  dns: [8.8.8.8]

  - name: data
    size: 100G
    mount: /var/data
boot:
  kernel: "uosc-kernel.elf"
  initrd: "omnisystem-initrd.img"
  cmdline: "console=virtio quiet"
features:
  deterministic: true
  nested_vaults: true
  record_replay: true
  live_migration: true
```

For containers:


## 5. Internal Architecture – Component Details

### 5.1 Environment Manager

- **State storage**: A content‑addressed directory in `audit-log` keeps the spec, current status, and resource usage.
- **Capability derivation**: For each environment, the manager requests a sub‑root capability from the host’s kernel. The derivation is bounded by the parent’s quota.
- **Lifecycle hooks**: Pre‑start (prepare volumes), post‑stop (clean up).

### 5.2 Payload Builder

Each environment type uses a different payload:

| Type | Payload | Implementation |
|------|---------|----------------|
| `vm` | UOSC kernel + initrd (or Linux kernel) | ELF loader that maps kernel into vault memory, sets up boot info |
| `container` | Rootfs + entrypoint process | Uses `sandbox` to spawn a process with isolated mount namespace, cgroups |
| `emulation` | QEMU binary + guest image | QEMU running as a process inside the vault, accessing virtio devices |
| `simulation` | BUSH simulator + model | BUSH binary with the model description, deterministic PRNG |

### 5.3 Virtio Device Providers

- **virtio-console**: Standard serial console for logs.
- **virtio-control**: For `build env exec` and guest agent communication; messages are protobuf‑encoded.
- **virtio-block**: Backed by a file or block device on the host. Supports snapshots (copy‑on‑write).
- **virtio-net**: Bridged to host network using `tap` or `macvtap`. QoS via TC.
- **virtio-vault**: Allows a guest to spawn nested environments. The host’s vault service validates the request against the guest’s quota.

### 5.4 Resource Accounting

- **Memory**: Hard limit enforced by Sanctum (hardware). The host also tracks usage for quota enforcement.
- **CPU**: Use CFS shares or real‑time throttling (via kernel).
- **Disk**: Quota via filesystem quotas or block device size limits.
- **Network**: Bandwidth limits via `tc` classful qdiscs.

### 5.5 Deterministic Mode & Record/Replay

- When `deterministic: true`, the environment is seeded with a fixed PRNG (provided at creation or start).
- All I/O events (keyboard, network, disk) are logged to the `audit-log` service, with a timestamp and a logical sequence number.
- To replay: `build env replay test --log-session <id>` – re‑runs the exact same input sequence and verifies the output matches.
- This is used for regression testing, debugging, and validation mesh.

### 5.6 Live Migration

**Prerequisites**: Shared storage or `p2p` streaming. The environment must be deterministic (no external nondeterminism) or use a checkpointable state.

**Procedure**:
1. `build env snapshot test --state-only` – capture memory and register state of the vault (excluding device state).
2. `p2p transfer` the snapshot to target host.
3. On target, create a new vault with the same capabilities and resources.
4. Restore memory and register state.
5. Resume execution from the point of snapshot.
6. Redirect network connections (via `conntrack` or virtual IP move).

**Post‑migration validation**: Run a short deterministic workload and compare outputs with the original (if both still running) to ensure fidelity.

### 5.7 Hot‑Reload of Environment Components

- The guest agent listens on `virtio-control` for `HotReload` requests.
- When received, it downloads the new binary from a content‑addressed CAS, verifies its signature and type compatibility, then triggers an atomic update using the kernel’s `inc-compile` service (if the component is a service) or a `kexec`‑style kernel update.
- The update is logged, and the rollback path is retained.

## 1. Fractional Resource Allocation and Over‑Commit Governance

Instead of static CPU and memory limits, the Environment Fabric introduces **fractional shares** and **soft/hard quotas** with predictable over‑commit behavior.

### 1.1 CPU Shares and Hard Limits

```yaml
resources:
  cpu:
    shares: 200          # proportional weight (default 1024)
    hard_limit: 4.5      # maximum cores (fractional)
    burst: 1.0           # extra burst capacity for <100ms
```

- **Shares** are enforced by the UOSC scheduler (CFS). A vault with shares=200 gets twice the CPU time of one with shares=100 when there is contention.
- **Hard limit** prevents any vault from exceeding a specified core count, even if idle. Fractional cores (e.g., 0.5) map to a single vCPU that is time‑sliced.
- **Burst** allows a vault to temporarily exceed its hard limit for short intervals, improving responsiveness for bursty workloads.

### 1.2 Memory with Compression and Ballooning

```yaml
resources:
  memory:
    hard_limit: 2048       # MB
    soft_limit: 1536       # MB – host attempts to reclaim above this
    balloon: true          # enable virtio‑balloon
    compression: zstd      # compress cold pages in the vault
```

- The vault manager can reclaim memory from a guest using virtio‑balloon, pressing the guest’s allocator to free pages.
- Cold pages are transparently compressed with `zstd` and stored in a host‑side swap area, invisible to the guest.

### 1.3 Over‑Commit Policy

Administrators define a **cluster over‑commit ratio** (e.g., memory 1.5×, CPU 4×). The scheduler monitors actual usage and automatically migrates vaults if a host becomes over‑committed, using live migration. This is governed by a deterministic rule set; AI can recommend adjustments but is not required.

## 2. Advanced I/O Virtualization

The Environment Fabric does not stop at virtio; it introduces **hardware‑backed I/O virtualization** for near‑native performance.

### 2.1 SR‑IOV and GPU‑mediated Passthrough

- For network and GPU, the fabric supports **SR‑IOV** (Single Root I/O Virtualization). A physical NIC or GPU is partitioned into Virtual Functions (VFs), and each VF is assigned directly to a vault via a hardware capability.
- **Fractional GPU (MIG)** : NVIDIA Multi‑Instance GPU allows a single GPU to be partitioned into multiple isolated slices. The `resources.gpu` stanza in the spec requests a slice (e.g., 1g.5gb).
- **GPU time‑sharing**: When MIG is not available, the `sandbox` service provides a **GPU scheduler** that interleaves kernel submissions from different vaults, enforcing quotas with microsecond granularity.

### 2.2 Persistent Memory and CXL

- Vaults can be given direct access to **persistent memory regions** (e.g., NVDIMM, CXL‑attached memory) as a `persistent_memory` resource.
- This enables databases or file systems inside the vault to run with zero‑copy persistence.

### 2.3 Unified Memory with GPU

- The USOS unified memory manager (already designed) allows vaults to allocate pages that seamlessly migrate between CPU RAM and GPU VRAM. The vault’s resource specification can include a `gpu_memory` quota that maps to the same unified pool.

## 3. Recursive Snapshot Trees and Instant Forking

Snapshots become the fundamental unit of environment lifecycle, not just an afterthought.

### 3.1 Snapshot Trees

Every environment can be **forked** (inspired by `memfork`). The result is a tree of related environments:

```bash
build env fork prod-db --name test-db --copy-on-write
```

- The child vault shares all read‑only pages with the parent (copy‑on‑write). Only diverging writes consume new memory.
- This enables instant cloning of production environments for testing, without duplicating data.

### 3.2 Recursive Snapshots

A snapshot captures not just the top‑level vault but **all nested vaults recursively**, as a single consistent point‑in‑time image. This is possible because nested vaults are children of the parent’s capability tree. The snapshot includes the full capability hierarchy and can be restored as a whole or drilled into to extract a specific nested vault.

### 3.3 Time‑Travel Branches

A snapshot can be tagged and used as a branch point. A developer can “go back” and replay a different sequence of inputs from that point, comparing the results.

```bash
build env branch prod-db --from-snapshot before-upgrade
```

The UVM can automatically fuzz both branches to check behavioral equivalence.

## 4. Universal Guest Agent Protocol

A single, lightweight agent binary (written in Titan) runs inside every environment, regardless of type. It implements a **gRPC‑like protocol over virtio‑control** with the following services:

- **Exec**: Execute commands, stream stdout/stderr back.
- **FileSystem**: Transfer files, mount volumes.
- **Metrics**: Expose CPU, memory, I/O, GPU usage.
- **HotReload**: Update kernel, services, or the agent itself.
- **NestedVault**: Create, list, destroy nested vaults (via the guest’s own `build env`).
- **Snapshot**: Trigger a guest‑coordinated filesystem freeze for consistent snapshots.

The agent is sandboxed with minimal capabilities, and its communication is encrypted and signed.

## 5. Deterministic Execution – Full System Replay

Deterministic mode captures **every non‑deterministic input** into a log, enabling perfect replay even for multi‑vCPU guests.

### 5.1 Capturing Non‑Determinism

- **Explicit I/O**: Keyboard, mouse, disk writes, network packets – logged at the virtio device level.
- **Implicit nondeterminism**: Interrupt arrival times, DMA completion order, and CPUID results are fixed using a virtual machine extension that intercepts and records these events.
- **Time**: The guest’s time perception is virtualised; the host injects a deterministic virtual clock.

### 5.2 Replay with Verification

- During replay, the recorded input log is fed in, and the output (console, network, disk writes) is compared against the original log. Any divergence immediately flags a regression.
- The UVM uses this to continuously validate that OS updates, driver conversions, or application changes do not alter behavior.

---

## 6. Live Migration – Zero‑Downtime, Cross‑Cluster

Live migration is not an afterthought; it is built into the vault’s capability model.

### 6.1 Pre‑Copy with Post‑Copy Fallback

- The migration daemon transfers memory pages iteratively while the vault continues to run. Dirty pages are retransmitted until a convergence threshold is reached.
- If pre‑copy cannot converge, the system switches to **post‑copy**: the vault is briefly paused, only CPU state is transferred, and the rest of memory is fetched on demand from the source host over the `p2p` fabric.

### 6.2 Device State Migration

- **SR‑IOV VF** can be hot‑unplugged and reassigned on the target host, with minimal packet loss.
- **GPU state** can be migrated by capturing the GPU context (using vendor APIs like NVIDIA’s MIG manager) or by replaying command buffers.

### 6.3 Multi‑Vault Transaction

A `build env` command can migrate an entire hierarchy (parent + nested) atomically, with a single migration token. All nested vaults are paused, snapshotted, transferred, and resumed in a coordinated manner, maintaining the tree structure.

## 7. AI‑Optional Predictive Scaling

An optional AI advisor (feature‑gated, shadow mode) observes historical resource usage across the cluster and predicts when a vault will need to scale. It can suggest:

- **Vertical scaling**: Increase memory or CPU shares before the guest hits its limit.
- **Horizontal scaling**: Fork the vault into a replica set behind a load balancer.
- **Pre‑emptive migration**: Move a vault to a host that is predicted to have lower load or better hardware affinity.

All suggestions are advisory; the deterministic core enforces quotas and never grants resources beyond the cluster capacity. The advisor’s decisions are logged and audited.

## 8. Formal Verification of the Fabric Itself

The entire Environment Fabric is built on the Omnisystem’s capability discipline. Axiom proofs guarantee:

- **Isolation**: A vault cannot access another vault’s memory, I/O, or capabilities.
- **Resource accounting**: The sum of allocated resources never exceeds the host’s capacity, and fractional allocation converges under over‑commit.
- **Snapshot consistency**: A recursive snapshot captures a consistent state (i.e., no in‑flight I/O that would corrupt the image).
- **Replay fidelity**: Replaying a recorded log produces exactly the same sequence of guest‑visible outputs.

These proofs are part of the CI pipeline; any code change that breaks a proof is rejected.
